Cómo usar este documento
Este documento es la guía maestra de construcción del sistema LUMA STORE SYSTEM. Está diseñado para que cualquier desarrollador, sea humano o IA, pueda seguirlo de principio a fin y construir el sistema completo sin ambigüedades.
Reglas fundamentales antes de empezar
Leer primero el Documento de Planeación del Sistema completo antes de escribir una sola línea de código. Ese documento describe QUÉ se construye. Este documento describe CÓMO y EN QUÉ ORDEN construirlo.
Las fases son secuenciales. No se debe avanzar a la siguiente fase sin haber completado y validado la anterior. Cada fase construye sobre la anterior.
Al finalizar cada fase existe una Lista de Validaciones. Se deben cumplir TODAS las validaciones antes de pasar a la siguiente fase. No hay excepciones.
Nunca eliminar ni refactorizar código de una fase anterior sin entender completamente cómo ese código es usado por las fases siguientes.
Cada módulo del backend (Django) se construye completo (modelos, serializers, vistas, URLs) antes de conectarlo al frontend (React).
El frontend nunca debe contener lógica de negocio. Toda la lógica vive en el backend. El frontend solo muestra datos y envía peticiones.

1.4 Base de datos en desarrollo — SQLite
En desarrollo se usa SQLite, que viene incluido con Python y Django. No requiere instalar nada ni crear una base de datos manualmente. Django crea el archivo db.sqlite3 en la raiz del proyecto automaticamente al correr las migraciones. La configuracion en settings.py ya lo maneja por defecto:
# settings.py — configuracion de base de datos para desarrollo (SQLite)
DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.sqlite3",
        "NAME": BASE_DIR / "db.sqlite3",
    }
}  # Esta es la config por defecto de Django. No hay que instalar nada extra.
El archivo db.sqlite3 debe agregarse al .gitignore para no subir datos de desarrollo al repositorio.

FASE 2  Modelos de Base de Datos
Definir y crear TODOS los modelos del sistema antes de construir ninguna vista. Esta es la fase más crítica del proyecto.
⚠️  IMPORTANTE: Esta fase define la estructura de datos completa del sistema. Una vez que las fases siguientes construyan sobre estos modelos, cambiarlos se vuelve costoso. Tomarse el tiempo necesario para hacerlo bien desde el inicio.
Todos los modelos deben crearse en sus respectivas apps de Django. Cada modelo es una tabla en la base de datos. Se presentan en orden de dependencia: primero los modelos que no dependen de otros, luego los que los referencian.
2.1 App: users — Usuarios y Roles
Archivo: backend/apps/users/models.py
from django.contrib.auth.models import AbstractUser
from django.db import models
class User(AbstractUser):
    """
    Extiende el usuario de Django con el campo de rol.
    AbstractUser ya incluye: username, email, first_name, last_name,
    password, is_active, date_joined.
    """
    class Role(models.TextChoices):
        OWNER  = "owner",  "Dueño"
        ADMIN  = "admin",  "Administrador"
        SELLER = "seller", "Vendedor"
        VIEWER = "viewer", "Visor"
    role = models.CharField(
        max_length=10,
        choices=Role.choices,
        default=Role.SELLER
    )
    # OBLIGATORIO: decirle a Django que use este modelo como AUTH_USER_MODEL
    # En settings.py agregar: AUTH_USER_MODEL = "users.User"
    def __str__(self):
        return f"{self.get_full_name()} ({self.role})"
⚠️  IMPORTANTE: En settings.py agregar AUTH_USER_MODEL = "users.User" ANTES de la primera migración. Si se hace después, las migraciones fallarán y habrá que resetear la base de datos.

class Category(models.Model):
    name        = models.CharField(max_length=100)
    description = models.TextField(blank=True)
    parent      = models.ForeignKey(
        "self", null=True, blank=True,
        on_delete=models.SET_NULL, related_name="subcategories"
    )
    order       = models.PositiveIntegerField(default=0)
    is_active   = models.BooleanField(default=True)
    created_at  = models.DateTimeField(auto_now_add=True)
    class Meta:
        ordering = ["order", "name"]
        verbose_name_plural = "categories"
class Product(models.Model):
    class Status(models.TextChoices):
        ACTIVE   = "active",   "Activo"
        INACTIVE = "inactive", "Inactivo"
        OUT      = "out",      "Agotado"
    name          = models.CharField(max_length=200)
    sku_base      = models.CharField(max_length=50, unique=True, blank=True)
    description   = models.TextField(blank=True)
    category      = models.ForeignKey(
        Category, null=True, blank=True,
        on_delete=models.SET_NULL, related_name="products"
    )
    cost          = models.DecimalField(max_digits=12, decimal_places=2, default=0)
    price         = models.DecimalField(max_digits=12, decimal_places=2)
    margin        = models.DecimalField(max_digits=6, decimal_places=4, default=0)
    is_visible    = models.BooleanField(default=False)  # Visible en portal
    is_featured   = models.BooleanField(default=False)  # Destacado en portal
    status        = models.CharField(max_length=10, choices=Status.choices, default=Status.ACTIVE)
    min_stock     = models.PositiveIntegerField(default=3)  # Umbral de alerta
    created_at    = models.DateTimeField(auto_now_add=True)
    updated_at    = models.DateTimeField(auto_now=True)
    def save(self, *args, **kwargs):
        # Auto-generar SKU base si está vacío
        if not self.sku_base:
            import uuid
            self.sku_base = str(uuid.uuid4())[:8].upper()
        super().save(*args, **kwargs)
class ProductImage(models.Model):
    product  = models.ForeignKey(Product, on_delete=models.CASCADE, related_name="images")
    image    = models.ImageField(upload_to="products/")
    order    = models.PositiveIntegerField(default=0)
    is_main  = models.BooleanField(default=False)
    class Meta:

class ProductVariant(models.Model):
    product    = models.ForeignKey(Product, on_delete=models.CASCADE, related_name="variants")
    size       = models.CharField(max_length=20, blank=True)  # Talla: XS, S, M, L, XL, etc.
    color      = models.CharField(max_length=50, blank=True)  # Color: Negro, Blanco, etc.
    sku        = models.CharField(max_length=80, unique=True)
    barcode    = models.CharField(max_length=100, blank=True)
    price      = models.DecimalField(
        max_digits=12, decimal_places=2, null=True, blank=True,
        help_text="Si es null, usa el precio base del producto"
    )
    stock      = models.PositiveIntegerField(default=0)
    is_active  = models.BooleanField(default=True)
    def get_price(self):
        return self.price if self.price is not None else self.product.price
    def save(self, *args, **kwargs):
        if not self.sku:
            self.sku = f"{self.product.sku_base}-{self.size}-{self.color}".upper()
        if not self.barcode:
            import uuid
            self.barcode = str(uuid.uuid4().int)[:12]
        super().save(*args, **kwargs)

    @property
    def segment(self):
        """Calcula el segmento automáticamente según historial."""
        from django.utils import timezone
        from apps.sales.models import Sale
        now = timezone.now()
        last_90 = Sale.objects.filter(customer=self, created_at__gte=now - timezone.timedelta(days=90))
        last_60 = last_90.filter(created_at__gte=now - timezone.timedelta(days=60))
        last_15 = last_90.filter(created_at__gte=now - timezone.timedelta(days=15))
        if last_15.count() == 1 and self.created_at >= now - timezone.timedelta(days=15):
            return "new"
        if last_60.count() > 3:
            return "frequent"
        if last_60.count() >= 1:
            return "regular"
        return "inactive"

from rest_framework.permissions import BasePermission
class IsOwner(BasePermission):
    def has_permission(self, request, view):
        return request.user.is_authenticated and request.user.role == "owner"
class IsOwnerOrAdmin(BasePermission):
    def has_permission(self, request, view):
        return request.user.is_authenticated and request.user.role in ["owner", "admin"]
class IsOwnerAdminOrSeller(BasePermission):
    def has_permission(self, request, view):
        return request.user.is_authenticated and request.user.role in ["owner", "admin", "seller"]
class CanViewOnly(BasePermission):
    """Visor: solo puede hacer GET."""
    def has_permission(self, request, view):
        if not request.user.is_authenticated:
            return False
        if request.user.role == "viewer":
            return request.method in ["GET", "HEAD", "OPTIONS"]
        return True

import axios from "axios";
const client = axios.create({
  baseURL: import.meta.env.VITE_API_URL || "http://localhost:8000/api/v1",
});
// Interceptor de REQUEST: agrega el token JWT a cada petición
client.interceptors.request.use((config) => {
  const token = localStorage.getItem("access_token");
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});
// Interceptor de RESPONSE: si el token venció (401), redirige al login
client.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (error.response?.status === 401) {
      localStorage.removeItem("access_token");
      localStorage.removeItem("refresh_token");
      window.location.href = "/login";
    }
    return Promise.reject(error);
  }
);
export default client;

Las rutas del panel admin deben estar protegidas. Un usuario no autenticado debe ser redirigido al login. Un usuario autenticado no debe poder acceder a rutas fuera de su rol:
import { Routes, Route, Navigate } from "react-router-dom";
import { useAuth } from "../hooks/useAuth";
// Ruta protegida: solo usuarios autenticados
function PrivateRoute({ children }) {
  const { user } = useAuth();
  return user ? children : <Navigate to="/login" replace />;
}
// Ruta protegida por rol
function RoleRoute({ children, allowedRoles }) {
  const { user } = useAuth();
  if (!user) return <Navigate to="/login" replace />;
  if (!allowedRoles.includes(user.role)) return <Navigate to="/" replace />;
  return children;
}

La lógica de costo/margen/precio es la más delicada de este formulario. Se implementa con un useEffect que observa los tres campos:
const [cost, setCost]     = useState(0);
const [margin, setMargin] = useState(0);
const [price, setPrice]   = useState(0);
const [lastEdited, setLastEdited] = useState(null); // "cost","margin" o "price"
useEffect(() => {
  if (lastEdited === "cost" || lastEdited === "margin") {
    // Recalcular precio
    if (margin < 100 && margin >= 0) {
      const newPrice = cost / (1 - margin / 100);
      setPrice(parseFloat(newPrice.toFixed(2)));
    }
  } else if (lastEdited === "price") {
    // Recalcular margen
    if (price > 0 && cost >= 0) {
      const newMargin = ((price - cost) / price) * 100;
      setMargin(parseFloat(newMargin.toFixed(4)));
    }
  }
}, [cost, margin, price, lastEdited]);

export function buildOrderMessage(order, template) {
  const productLines = order.items.map(item =>
    `  - ${item.variant.product.name} talla ${item.variant.size} color ${item.variant.color} x${item.quantity} — $${item.subtotal.toLocaleString("es-CO")}`
  ).join("\n");
  return template
    .replace("{nombre_cliente}", order.customer_name || "Cliente")
    .replace("{numero_pedido}", order.number)
    .replace("{productos}", productLines)
    .replace("{total}", `$${order.total.toLocaleString("es-CO")}`)
    .replace("{estado}", order.status_display);
}

8.2 Estado del Carrito — CartContext.jsx
El carrito vive en un Context de React usando useReducer. El estado del carrito se persiste en localStorage para que no se pierda si el usuario cierra la pestaña accidentalmente:
const initialState = {
  items: JSON.parse(localStorage.getItem("cart_items") || "[]")
};
function cartReducer(state, action) {
  switch (action.type) {
    case "ADD_ITEM": {
      const exists = state.items.find(i =>
        i.variant_id === action.payload.variant_id
      );
      if (exists) {
        // Incrementar cantidad (sin exceder stock)
        const updated = state.items.map(i =>
          i.variant_id === action.payload.variant_id
            ? { ...i, quantity: Math.min(i.quantity + action.payload.quantity, i.max_stock) }
            : i
        );
        return { ...state, items: updated };
      }
      return { ...state, items: [...state.items, action.payload] };
    }
    case "REMOVE_ITEM":
      return { ...state, items: state.items.filter(i => i.variant_id !== action.payload) };
    case "UPDATE_QUANTITY":
      return { ...state, items: state.items.map(i =>
        i.variant_id === action.payload.variant_id
          ? { ...i, quantity: action.payload.quantity }
          : i
      )};
    case "CLEAR":
      return { ...state, items: [] };
    default:
      return state;
  }
}
// Sincronizar con localStorage en cada cambio
useEffect(() => {
  localStorage.setItem("cart_items", JSON.stringify(state.items));
}, [state.items]);

9.2 Cálculo de Predicción de Reabastecimiento
La lógica de predicción se calcula en el backend como parte de la respuesta del dashboard. Para cada variante activa:
from django.db.models import Sum
from django.utils import timezone
def calculate_restock_alerts():
    alerts = []
    thirty_days_ago = timezone.now() - timezone.timedelta(days=30)
    for variant in ProductVariant.objects.filter(is_active=True, stock__gt=0):
        # Unidades vendidas en los últimos 30 días
        units_sold = StockMovement.objects.filter(
            variant=variant,
            type="sale",
            created_at__gte=thirty_days_ago
        ).aggregate(total=Sum("quantity"))["total"] or 0
        if units_sold > 0:
            daily_avg = abs(units_sold) / 30
            days_remaining = variant.stock / daily_avg
            if days_remaining < 10:
                alerts.append({
                    "variant_id": variant.id,
                    "product_name": variant.product.name,
                    "size": variant.size,
                    "color": variant.color,
                    "current_stock": variant.stock,
                    "days_remaining": round(days_remaining, 1),
                })
    return sorted(alerts, key=lambda x: x["days_remaining"])

FASE 12  Preparación para Producción y Entrega
Configurar el sistema para producción, construir los builds de React y preparar el paquete de entrega al cliente.
12.1 Build de producción del frontend
En producción, Django sirve los archivos compilados de React como archivos estáticos. Esto significa un solo servidor para todo el sistema. El proceso es:
# Paso 1: Construir ambas apps React
cd frontend/admin && npm run build
# Esto genera frontend/admin/dist/
cd frontend/store && npm run build
# Esto genera frontend/store/dist/
# Paso 2: Copiar los builds al directorio de estáticos de Django
# En settings.py de producción:
STATICFILES_DIRS = [
    BASE_DIR / "frontend/admin/dist",
    BASE_DIR / "frontend/store/dist",
]
# Paso 3: Configurar Django para servir las apps React
# En config/urls.py agregar al FINAL (después de todas las rutas API):
from django.views.generic import TemplateView
urlpatterns += [
    # Portal de clientes en /tienda/
    path("tienda/", TemplateView.as_view(template_name="store/index.html")),
    path("tienda/<path:path>", TemplateView.as_view(template_name="store/index.html")),
    # Panel admin en / (raíz)
    re_path(r"^(?!api/)(?!admin/).*$", TemplateView.as_view(template_name="admin/index.html")),
]
12.2 Configuración de producción en settings.py
# settings_production.py (separar del settings de desarrollo)
from .settings import *
DEBUG = False
ALLOWED_HOSTS = ["tudominio.com", "www.tudominio.com"]
# Seguridad
SECURE_BROWSER_XSS_FILTER = True
SECURE_CONTENT_TYPE_NOSNIFF = True
X_FRAME_OPTIONS = "DENY"
# CORS solo permite el dominio del cliente
CORS_ALLOWED_ORIGINS = ["https://tudominio.com"]
# Archivos estáticos
STATIC_ROOT = BASE_DIR / "staticfiles"
python manage.py collectstatic --noinput

12.4 Proceso de instalación para un cliente nuevo
Cuando se vende el sistema a un cliente nuevo, el proceso de instalación sigue estos pasos exactos:
Clonar o copiar el repositorio base a un nuevo directorio con el nombre del cliente.
Crear la base de datos PostgreSQL nueva para este cliente en el servidor de produccion.
Copiar .env.example a .env y completar todos los valores con los datos del cliente nuevo.
Instalar dependencias: pip install -r requirements.txt.
Aplicar migraciones: python manage.py migrate.
Crear el usuario dueño: python manage.py createsuperuser.
Hacer el build del frontend: npm run build en ambas apps React.
Ejecutar python manage.py collectstatic.
Iniciar el servidor y verificar que todo funciona.
Entregar al cliente las credenciales del usuario dueño y el link al panel admin.
